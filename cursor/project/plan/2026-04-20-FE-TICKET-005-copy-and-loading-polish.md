# FE-TICKET-005: Copy and loading polish — Implementation Plan

## Overview

Implement **string-only** updates so the main flow’s hero, loading state, and result screens read consistently with the **three-class** model (`legit` / `warning` / `fraud`): clear about multiple outcomes, explicit where fraud differs from “needs review,” and calm in loading copy. **No API, routing, or type changes.**

**Dependency:** **FE-TICKET-002** should be done first so three views and `getRiskTier` are stable; this ticket only edits text on top of that structure.

## Desired End State

- **Hero and loading** do not read as a binary scam detector unless that is explicitly intentional; where helpful, copy acknowledges **multiple risk levels** (safe / review / fraud-tier) without alarming users.
- **Result screens** keep tier-specific badges but use a **neutral, shared page title** that does not imply a single “prediction class” only, **or** use **tier-specific titles** that match FE-TICKET-002 visuals — pick one approach and apply it consistently (see Phase 2).
- **Browser tab / SEO** metadata in `app/layout.tsx` matches Job Sentry and the risk-check purpose (not Create Next App).
- Verification: `npm run lint` and `npm run build` succeed; spot-check all three result tiers and loading in light/dark mode.

### Key Discoveries

- Nearly all strings for this ticket live in **`app/page.tsx`**; root **`metadata`** is the main out-of-file gap.
- **`ProbabilityBreakdown`** already surfaces all three classes; editorial focus is hero, loading arrays, shared `<h1>`, and CTA wording.

## What We're NOT Doing

- Changing **`lib/api`**, request/response types, or **`predictScam`** behavior.
- Changing **`getRiskTier`**, routing, or component structure (unless a **constant rename** is required for a label, e.g. exporting a shared string — keep diffs minimal).
- Redesigning **`LoadingFrame`** timing, animations, or **`estimateLoadingDurationMs`** math.
- Large extractions (e.g. moving all copy to `messages/en.json`) unless the team explicitly asks in a follow-up.

## Implementation Approach

**Chosen editorial defaults** (aligned with acceptance criteria; product can swap exact wording in the same locations):

1. **Hero:** Shift framing toward **risk / safety** while keeping the product recognizable — e.g. title **Job Post Risk Check** (or keep **Job Post Scam Check** if product wants legacy wording) plus a subtitle that states the model considers **legitimate, questionable, and high-risk** signals (or equivalent plain language).
2. **CTA:** Replace **Validate** with **Check post** or **Run check** to avoid implying a single pass/fail verdict.
3. **Loading:** Edit **`LOADING_STATUS_MESSAGES`** and **`LOADING_TIP_LINES`** in place — add or adjust **1–3 lines** so one message contrasts **review-level** cues vs **stronger fraud signals** without sensational language; keep length similar so layout does not break.
4. **Result `<h1>`:** Use one neutral title for all three views, e.g. **Risk check results** or **Your assessment**, instead of **Prediction Results** (which can read model-centric). Tier-specific meaning stays in badges (**Looks Safe** / **Review carefully** / **Fraud risk**).
5. **Layout metadata:** Set `title` and `description` to Job Sentry + short risk-check description.
